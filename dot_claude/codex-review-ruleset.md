# codex レビュー判定ルールセット

codex のレビュー出力（公式 codex プラグイン：review-gate / `/codex:review` / `/codex:adversarial-review`）を **取捨選択するときの判定基準**。
また `/codex:adversarial-review` に **focus テキスト**として渡せば、codex 自身にこの観点でレビューさせられる。

## ⚠️ 最重要：codex の過剰防御癖を打ち消す

codex は放っておくと「もっと防御コードを足せ」と提案しがち。**その方向の指摘は採用しない。**
むしろ逆に、**過剰防御そのものを YAGNI 違反として却下／除去候補**にする。却下した指摘は理由を添えてユーザーに伝える。

過剰防御として却下・除去対象にするもの：

- 到達し得ない / 呼び出し側で保証済みの null・undefined ガード
- 握りつぶす（再 throw も記録もしない）try-catch、意味のないフォールバック
- 使われていない・将来用の引数 / オプション / 抽象 / フック
- 不要な防御的コピー、過剰な存在チェック
- 仕様にない入力分岐・エラー分岐

## 採用する観点（この基準で取捨選択する）

1. **YAGNI / 過剰防御の排除** … 上記の通り。
2. **DRY** … 重複ロジック・重複定数・コピペ。既存の共通実装があるのに再発明している箇所。
3. **プロジェクト規約への準拠** … `~/.claude/CLAUDE.md`（global）とプロジェクト root の `CLAUDE.md`、`react-coding-rules` / `typescript-coding-rules` 等のルール。命名・export 方針・型推論・ディレクトリ構成・エラーハンドリング方針。
4. **de-facto 規約** … 同種の処理がリポ内の他所でどう書かれているか。新規 / 変更コードがその流儀と食い違っていないか。
5. **明白な重大バグ** … 未定義参照・型エラー・データ破壊やセキュリティのリスクは、観点に関わらず採用して対応する。

## 却下するもの

- 過剰防御の追加提案（最重要・上記）
- 好みの範疇の nitpick で明確な改善がないもの
- 既存コードや設計方針と矛盾する提案
- 周辺コンテキストを読めば不要とわかる、前提を誤解した指摘

## 適用方針

- 採用する指摘のうち、リスクが低く意図が明確なものは修正を適用する。
- 設計判断を伴う・トレードオフがある・影響範囲が広いものは、適用せずユーザーに報告して判断を仰ぐ。
- 判定は必ず該当ファイルの現行コードを読んでから行う（codex の要約だけで判断しない）。
