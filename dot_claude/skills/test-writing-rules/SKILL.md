# テスト作成ルール

## 大原則

- テストは **「退行検知」と「仕様の言語化」** のために書く
- カバレッジ閾値は退行検知の最低ライン。閾値を稼ぐためのテストは書かない
- テストの本質は仕様の検証。実装をそのまま assert に翻訳した「tautological test」は禁止（実装にバグがあっても通るため）
- 順序は **仕様 → 観点導出 → 実装を読んで漏れ補強**。逆にしない

### FIRST 原則

| 項目 | 意味 |
| --- | --- |
| **Fast** | ms オーダー。遅いと回さなくなる |
| **Independent** | 実行順・他テスト状態に依存しない（共有 DB / グローバル変数の汚染禁止） |
| **Repeatable** | 何度回しても同じ結果（時刻・乱数・ネットワークは制御下） |
| **Self-validating** | 合否を assert で自動判定。目視確認テストは作らない |
| **Timely** | 実装と同時に書く。後追いだと実装に引きずられ tautological になる |

> Red/Green/Refactor のサイクル運用自体は `tdd` skill を参照。本 skill はケース設計に特化。

## テストレベル戦略

各レベルの目的が違う。**どのレベルで何を保証するか** を意識して二重テストを避ける:

| レベル | 目的 | 速度 | 数の目安 |
| --- | --- | --- | --- |
| **単体** | ロジック・分岐の網羅 | 最速 | 多 |
| **結合** | モジュール境界・I/O 契約 | 中 | 中 |
| **E2E** | ユーザーシナリオ・本番に近い経路 | 遅 | 少 |

- ドメインロジックの分岐網羅は **単体** で済ませる。E2E に押し付けない
- モジュール間の契約（型・スキーマ・呼び出し回数）は **結合** で
- ユーザーゴールに直結するシナリオのみ **E2E** で 1〜2 本

> Testing Trophy の発想: 結合（integration）の比重を上げ境界バグを拾う。HTTP は MSW、DB はテストコンテナ、外部 SDK のみモックが定石。

## ワークフロー

テストを書く前に毎回辿る:

1. **仕様の読解** — 対象のシグネチャ・型・ドキュメント・呼び出し元のユースケースを把握
2. **観点の列挙** — 下記の 3 軸を当ててケース候補を出す（詳細: `references/perspectives.md`）
3. **技法の選択** — 関連する技法を選んでケースに展開（詳細: `references/techniques.md`）
4. **実装で漏れ補強** — 列挙したケースが全分岐を踏めているか確認、足りなければ仕様視点で追加
5. **記述** — 1 ケース 1 テストに落とす。適用した技法名は短くコメント or describe 名に残す

## 観点を導出する 3 軸

書き始める前に各軸で漏れを潰す。各軸の項目展開・ドメイン別チェックリスト・具体例は `references/perspectives.md`。

- **軸 1: 入出力の分類** — 正常系 / 準正常系（境界・空・特殊値・Unicode・時刻）/ 異常系（エラーの型・メッセージまでアサート、Result でも throw でも形式は問わない）/ 例外系（前提崩壊で起きる throw）
- **軸 2: 機能横断** — 冪等性 / 並行性 / 順序依存 / タイムアウト・リトライ / 部分失敗 / リソースリーク / 入力サニタイズ / i18n・エンコーディング / 観測性
- **軸 3: ドメイン特有** — LLM / Agent / Cloudflare ランタイム / ブラウザ / バックエンド API / バッチ など、対象に応じて当てる

## ミューテーション思考

書いたケースが効くか、実装を頭の中で mutate（`>` → `>=`、`&&` → `||`、`return x` → `return undefined` など）して落ちるか確認する。落ちないテストは仕様ではなく実装を写しているだけの可能性が高い。詳細・mutation 表は `references/perspectives.md`。

## 構造

- AAA（Arrange / Act / Assert）の流れは大半の単体テストで自然に出る。明示コメント・空行は **任意**。冗長なら入れない
- GWT（Given / When / Then）は受け入れテストや複雑な業務シナリオに限って使う
- テスト名は「対象 / 条件 / 期待振る舞い」の 3 要素  
  例: `parseConfig: schema 違反の YAML を渡すと ValidationError を throw する`
- `describe` のネストは 2 階層まで。深くなるならファイルを分ける
- 1 テスト 1 検証。複数アサートを 1 テストに詰めない（境界値の代表値はケースを分割）

## テストダブル

「モック」は広義/狭義が混在しやすい。意図して使い分ける:

| 種類 | 役割 |
| --- | --- |
| **dummy** | 引数を埋めるだけ、呼ばれない |
| **stub** | 固定の戻り値を返す（状態検証） |
| **spy** | 呼び出し記録を取れる stub |
| **mock** | 期待した呼び出しが起きたかを検証（振る舞い検証） |
| **fake** | 軽量実装（in-memory DB、in-memory KV など） |

ルール:

- モック化は **I/O 境界に限定**（HTTP / DB / 時刻 / 乱数 / ファイル / 外部 API）。ドメインロジック自体や自分のコードはモックしない
- 時刻依存は `vi.useFakeTimers()` または DI した clock。`new Date()` 直書きはラップ提案を出す
- モック戻り値は **本番スキーマに準拠**（プロジェクトの Zod / valibot / 型ガードを通る形）
- 外部 API は **契約テスト**（Pact / OpenAPI スキーマ突合等）で実体との乖離を検出。モックだけでは「モックが古い」事故を防げない

## テストデータ

- リテラル直書きは小さなテストでのみ。複雑な構築は **Test Data Builder / Object Mother** に切り出す
  - 例: `aUser().withRole("admin").withMfa(true).build()`
- 「テストごとに必要な差分だけを書く」設計にする。共通項を builder のデフォルトに寄せる
- fixture ファイル（JSON 等）は **何のための fixture か** を名前に込める。`user.json` ではなく `user-without-email.json`
- ランダム生成（faker / fast-check）は **seed 固定 or 失敗時 seed のログ保存** を必須に。Flaky の温床

## Flaky テスト対策

落ちたり通ったりするテストは **0 件を維持**。発見したら即対処、修正不能なら隔離（skip ではなく別 suite に追い出す）して期日を切る。

主因と対策:

- **時刻依存** → fake timer / 注入 clock
- **乱数依存** → seed 固定
- **ネットワーク・I/O 競合** → モック or 専用環境、テストごとに DB をリセット
- **非同期待ち** → `sleep(N)` 禁止。`waitFor` / イベント駆動の待機を使う
- **テスト間の状態共有** → `beforeEach` で必ずクリーン化、グローバル変数を持たない
- **順序依存** → `--shuffle` で実行して通ることを確認

## 禁止事項 (Test Smells)

- **tautological test** — 実装をそのまま assert に翻訳しただけ
- **private 状態の覗き見** — リフレクションで内部状態を直接 assert
- **巨大スナップショット** — 差分が読めない / 何が壊れたか分からないもの
- **placeholder** — `expect(true).toBe(true)` 系
- **skip / only のコミット** — `it.only` を残してマージしない
- **mystery guest** — テストファイル外のデータ（共有 DB / 外部ファイル）に依存して何が前提か読めない
- **conditional logic** — テスト内に `if` を書かない（分岐するならケースを分ける）
- **slow test** — 1 ケース秒オーダーになったら設計を疑う
