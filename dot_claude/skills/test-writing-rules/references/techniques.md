# テスト技法カタログ

観点（`perspectives.md`）で出したケース候補に最適な技法を当てて具体ケースへ落とすためのカタログ。

## 1. 同値分割 (Equivalence Partitioning)

### 概要

入力空間を「同じ振る舞いをするはず」のグループに分け、各グループから 1 つだけ代表を選ぶ。

### 適用シーン

- 入力が連続値・離散値どちらでも、振る舞いが分岐する区間が明確
- 全入力を試すコストが見合わない

### 例

「年齢で割引率を決める関数」(0–5: 無料 / 6–17: 50% / 18–64: なし / 65+: 30%):

- 各区間から代表 1 つ: 3, 12, 30, 70 → 4 ケース
- 全年齢 0–120 を試す必要はない

## 2. 境界値分析 (Boundary Value Analysis)

### 概要

同値分割の補完。区間の **境界そのもの・境界 ± 1** を必ず試す。バグは境界に集中する。

### 適用シーン

- if / switch で範囲分岐がある
- 上限・下限・配列長・文字列長

### 例

上記の年齢関数なら: 0, 5, 6, 17, 18, 64, 65, -1（範囲外） を追加。同値分割と組み合わせて使う。

### よくある境界

- 数値: `MIN`, `MIN+1`, `MAX-1`, `MAX`, `0`, `-1`
- 配列: `[]`, `[x]`, `[x, y]`, `length === capacity` 直前
- 文字列: `""`, `" "`, 1 文字, 仕様の最大長, 最大長 +1

## 3. デシジョンテーブル (Decision Table)

### 概要

複数の条件の組合せで結果が決まるロジックを表に展開し、行ごとに 1 ケース化する。

### 適用シーン

- 複数フラグで分岐するビジネスルール
- 条件が `if (A && B && !C)` のように絡む
- 仕様書がそもそもテーブルで書かれている

### 例

「会員」「セール期間」「クーポン」で割引率が決まるとき:

| 会員 | セール | クーポン | 期待割引 |
| --- | --- | --- | --- |
| ◯ | ◯ | ◯ | 40% |
| ◯ | ◯ | × | 30% |
| ◯ | × | ◯ | 20% |
| × | ◯ | ◯ | 25% |
| × | ◯ | × | 15% |
| × | × | × | 0% |

組合せが爆発する場合はペアワイズ（後述）に切り替える。

## 4. 状態遷移テスト (State Transition Testing)

### 概要

状態を持つ対象の「状態 × イベント → 次状態」を表に書き、各遷移を 1 ケース化する。**不正な遷移** （起こり得ないイベント）も明示的にテスト。

### 適用シーン

- ステートマシン、ワークフロー
- フォームのバリデーション・送信フロー
- 認証フロー（未ログイン → ログイン中 → 期限切れ）
- 注文ステータス（pending → paid → shipped → delivered / cancelled）

### 例

注文ステータス:

| 現状態 | イベント | 次状態 | テストすべきか |
| --- | --- | --- | --- |
| pending | pay | paid | ◯ |
| pending | cancel | cancelled | ◯ |
| paid | ship | shipped | ◯ |
| paid | cancel | refunding | ◯ |
| shipped | cancel | （不可） | ◯ エラー確認 |
| delivered | pay | （不可） | ◯ エラー確認 |

## 5. ペアワイズ (Pairwise / All-pairs)

### 概要

全組合せを試すと爆発するパラメータ群に対し、「**全ての 2 因子の組合せ** が少なくとも 1 ケースに現れる」よう削減する。多くのバグは 2 因子の組合せで再現するという経験則に基づく。

### 適用シーン

- パラメータが 4 つ以上、各 2–4 値
- 全組合せが数十〜数百になり現実的でない
- ブラウザ × OS × 言語 × プラン などのマトリクス

### 例

ブラウザ(3) × OS(3) × プラン(2) × 言語(2) = 36 組合せ → ペアワイズなら 9 程度。
生成ツール: PICT (Microsoft), `pict-node`, `allpairspy` など。

## 6. エラー推測 (Error Guessing)

### 概要

経験から「ここでバグが出やすい」を狙い撃ちする。形式化しにくいが効果が大きい。チェックリスト化して持つ。

### 狙い目チェックリスト

- 空入力（`""`, `[]`, `{}`, `null`, `undefined`, `NaN`, `0`）
- 1 / 2 要素の境界
- 重複・並び順
- 同時実行・連打
- 大文字小文字差・全角半角差
- 巨大入力 / 0 件入力
- 過去日 / 未来日 / うるう年 / TZ 跨ぎ
- 逆順・降順
- 前ステップが失敗した状態
- ネットワーク切断・タイムアウト
- 権限なし / 期限切れトークン

### 適用シーン

他の技法で機械的に出ないケースを補完するときに使う。最後に必ず 1 周当てる。

## 7. プロパティベーステスト (Property-based Testing)

### 概要

「具体例」ではなく「**入力が満たすべき性質**」を宣言し、フレームワークがランダム入力を大量に生成して検証する。失敗時は最小ケースに自動縮約 (shrinking) される。

### 適用シーン

- 純粋関数・データ変換・パーサー / シリアライザ
- 可換律・結合律・冪等性などの代数的性質が言える対象
- エッジケースを人間が網羅しきれない領域

### 例 (fast-check)

```ts
import fc from "fast-check";

// 性質: parse(stringify(x)) === x （ラウンドトリップ）
test("JSON roundtrip", () => {
  fc.assert(
    fc.property(fc.jsonValue(), (v) => {
      expect(JSON.parse(JSON.stringify(v))).toEqual(v);
    })
  );
});

// 性質: sort は冪等
test("sort is idempotent", () => {
  fc.assert(
    fc.property(fc.array(fc.integer()), (arr) => {
      const once = [...arr].sort((a, b) => a - b);
      const twice = [...once].sort((a, b) => a - b);
      expect(twice).toEqual(once);
    })
  );
});
```

### よく使う性質

- **ラウンドトリップ**: `decode(encode(x)) === x`
- **冪等性**: `f(f(x)) === f(x)`
- **可換性**: `f(a, b) === f(b, a)`
- **結合性**: `f(f(a, b), c) === f(a, f(b, c))`
- **不変条件**: 「並べ替えても要素数は変わらない」
- **メタモルフィック**: 入力を a → b と変えたとき、出力が決まった関係を保つ

## 技法の組合せ早見

| 状況 | 第一選択 | 補完 |
| --- | --- | --- |
| 単純な if 分岐 | 同値分割 | 境界値 |
| 数値範囲 | 境界値 | 同値分割 |
| 複数フラグ AND/OR | デシジョンテーブル | ペアワイズ |
| 状態遷移 | 状態遷移テスト | エラー推測 |
| パラメータ多数 | ペアワイズ | エラー推測 |
| 純粋関数・変換 | プロパティベース | 境界値 |
| UI ワークフロー | 状態遷移 | エラー推測 |

最後に **エラー推測** を 1 周回すと、機械的技法では拾えないドメイン固有の罠を拾える。
